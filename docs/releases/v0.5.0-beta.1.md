# sp_bridge v0.5.0-beta.1 Release Notes

> 發布日期：2026-03-28
> 分支：`claude/fivem-bridge-review-1k1jl`
> 前一版本：v0.3.0（initial release）

---

## 這個版本做了什麼

v0.5.0-beta.1 是 sp_bridge 第一個正式 Beta 版本。相較於 v0.3.0，本版本完成了以下工作：

### 1. Framework Provider 架構化（Sprint 2–8）

- 引入 `sp.frameworkProvider` provider 模式，ESX / QBCore / QBOX 各有獨立 `providers/` 實作。
- 所有框架相關 API（player data、job、money、character id）全面路由至 provider，不再直接呼叫 `CoreObject`。
- ESX `GetPlayerFromId` 改以 `exports.es_extended:GetPlayerFromId` 為主路徑，解決 `CoreObject` 過時造成的 nil 問題。

### 2. Raw / Normalized API 邊界確立（Sprint 3）

- 新增 `GetNormalizedPlayerData`、`GetNormalizedJob`、`GetNormalizedMoney`：回傳跨框架統一 schema。
- `GetPlayerJob` 改為 `GetNormalizedJob` 的 alias（**重大變更，見下方**）。
- 新增 `GetRawPlayerJob`：回傳框架原始 job 物件。
- 新增 `GetFrameworkIdentifier`：回傳平台 identifier（`license:xxx`），不與角色 ID 混淆。
- `GetCharacterId` 為正式 canonical 角色 ID export；`GetCitizenId` 保留為 legacy alias。

### 3. Inventory Provider 架構化（Sprint 5–6）

- 引入 `sp.inventoryProvider`，ox / qb / qs 各有獨立 provider 實作。
- 7 個 inventory server export 全面路由至 provider。
- ESX 無第三方 inventory 時，`AddItem` 有 native fallback（`Player.addInventoryItem`）；`CreateUsableItem` 有三段 fallback。

### 4. Banking Provider 架構化（Sprint 9–9.1）

- 引入 `sp.bankProvider`，新增 4 個 provider：`framework`、`Renewed-Banking`、`qb`、`okokBanking`。
- 新增 6 個 server exports：
  - `GetPlayerBankBalance(source)`
  - `AddPlayerBankMoney(source, amount, reason)`
  - `RemovePlayerBankMoney(source, amount, reason)`
  - `GetSocietyBalance(accountId)`
  - `AddSocietyMoney(accountId, amount, reason)`
  - `RemoveSocietyMoney(accountId, amount, reason)`
- 新增 `GetBankProviderName()` 供診斷使用。
- Provider 載入時印出 capability log：`[banking] provider=xxx playerBank=true/false society=true/false`。
- 所有 banking wrapper 在失敗時印出 structured failure log（含 reason code）。

### 5. 自動測試資源（Sprint 8–10）

- 新增 `sp_bridge_test` FiveM resource，指令 `sp_test` / `sp_test <src>` 觸發測試。
- 涵蓋：framework detection、player data（normalized / raw）、inventory、banking。
- 測試設計為 provider-agnostic：不同框架組合下通過率可能不同，但不應出現 SCRIPT ERROR。

## 重大變更

### `GetPlayerJob` 現在回傳 NormalizedJob

**v0.3.0 行為**：回傳框架原始 job 物件（ESX 為 `Player.job` table，QB 為 `pd.job` table）。

**v0.5.0-beta.1 行為**：回傳統一 NormalizedJob schema：

```lua
{
    name    = string,
    label   = string,
    onduty  = boolean,
    isBoss  = boolean,
    grade   = {
        level  = number,
        name   = string,
        label  = string,
        salary = number,
    }
}
```

**升級方式**：
- 若需要原始 job 物件，改用 `GetRawPlayerJob(source)`。
- 若需要 normalized schema，`GetPlayerJob` 已可直接使用，不需改動。
- `job.isboss`（小寫）、`job.grade_name`、`job.grade_label` 等舊欄位已不存在於 normalized schema。

- `GetNormalizedJob(source)`
- `GetNormalizedMoney(source)`

## 舊 API 保留狀態

| API | 狀態 | 建議替換 |
|-----|------|---------|
| `GetCitizenId(source)` | legacy alias，保留 | 改用 `GetCharacterId` |
| `GetPlayerData(source)` | raw / legacy-compatible，保留 | 改用 `GetNormalizedPlayerData` |
| `GetBankBalance(source)` | 舊 bank module export，保留 | 改用 `GetPlayerBankBalance` |
| `AddBankMoney(source, amount, reason)` | 舊 bank module export，保留 | 改用 `AddPlayerBankMoney` |
| `RemoveBankMoney(source, amount, reason)` | 舊 bank module export，保留 | 改用 `RemovePlayerBankMoney` |

> 舊 API 在 Beta 1 中不會主動移除，但新開發建議一律使用 canonical / normalized API。

## 已知限制

詳見 `docs/known-limitations.md`。主要限制摘要：

1. Client side API 完整度低於 server side。
2. Inventory metadata 支援程度因 provider 版本而異。
3. `framework` banking provider 不支援 society account。
4. `esx_banking` / `esx_addonaccount` 偵測但無 provider 實作。
5. Notification / Target 模組尚未標準化。

## 如何測試

### 1. 啟動 `sp_bridge_test`

在 `server.cfg` 中加入（排在 `sp_bridge` 之後）：

```
ensure sp_bridge
ensure sp_bridge_test
```

### 2. 執行測試

```
# 不帶 src：跳過所有需要線上玩家的測試
sp_test

# 帶 src：執行完整測試套件（替換 1 為實際玩家 server id）
sp_test 1
```

### 3. 解讀結果

- `[PASS]`：預期行為確認
- `[FAIL]`：非預期行為，需修復
- `[SKIP]`：條件不滿足（無玩家 / provider 不支援）——不代表錯誤

---

## 升級方式

從 v0.3.0 升級：

1. 替換 `sp_bridge` 資源資料夾。
2. 確認 `config.lua` 設定（新增 `Config.Banking`，預設為 `auto-detect`）。
3. 搜尋並替換以下 API 呼叫（非強制，但建議）：
   - `GetCitizenId` → `GetCharacterId`
   - `GetPlayerData`（若依賴正規化欄位）→ `GetNormalizedPlayerData`
4. 若有依賴 `GetPlayerJob` 回傳的原始 job 欄位（如 `job.isboss`），改用 `GetRawPlayerJob`。
5. 執行 `sp_test <src>` 確認無 FAIL。
